## Canonical Layout

```
my-ls-usecase/
│
├── backend/                          # Backend API Lambda handlers — one file per endpoint
│   ├── get_products.py
│   ├── get_sales_summary.py
│   ├── get_anomalies.py
│   └── get_trend_analysis.py
│
├── terraform/                        # All IaC — one .tf file per concern
│   ├── main.tf                       # Provider block + core auth / UI resources
│   ├── versions.tf                   # Provider version constraints
│   ├── variables.tf                  # Input variables (project_name, region, env …)
│   ├── locals.tf                     # Derived names computed from project_name
│   ├── outputs.tf                    # Terraform outputs consumed by scripts & UI
│   ├── data_infrastructure.tf        # S3, Glue, DynamoDB, Athena
│   ├── backend_lambdas.tf            # API Lambda functions + packaging
│   ├── agent_lambdas.tf              # Agent tool Lambda functions
│   ├── report_lambda.tf              # Report generation Lambda trio (initiator/worker/status)
│   ├── report_ssm_parameters.tf      # SSM parameters for report job config
│   ├── api_gateway.tf                # API Gateway routes + integrations
│   └── sql/                          # Athena view SQL files (executed post-apply)
│       ├── monthly_sales_view.sql
│       └── anomaly_detection_view.sql
│
├── data/                             # Source data — SEPARATE from terraform/
│   ├── csv/                          # Raw CSV data files
│   ├── parquet/                      # Converted Parquet files (S3 upload target)
│   └── csv_to_parquet.py             # One-off conversion utility
│
├── agents/
│   ├── trend-analysis-agent/         # One folder per agent
│   │   ├── agent/                    # Agent container entry point
│   │   │   ├── trend_analysis_agent.py  # Agent code (Strands + BedrockAgentCoreApp)
│   │   │   ├── requirements.txt      # Python deps for the agent container
│   │   │   └── .bedrock_agentcore.yaml  # Generated by agentcore configure
│   │   ├── tools/                    # Lambda implementations of each agent tool
│   │   │   ├── get_sales_by_region.py
│   │   │   └── get_trend_metrics.py
│   │   └── schemas/                  # JSON schemas for gateway registration
│   │       ├── get-sales-by-region.json
│   │       └── get-trend-metrics.json
│   └── anomaly-detection-agent/
│       ├── agent/
│       ├── tools/
│       └── schemas/
│
├── report_lambda/                    # Async long-running report Lambdas (3-file pattern)
│   ├── initiator.py                  # Receives request, creates job, invokes worker async
│   ├── worker.py                     # Runs the actual computation (can take 60-120 s)
│   └── status_checker.py             # Polls DynamoDB job table, returns result when ready
│
├── agentcore_gateway/                # Gateway setup scripts (run post-terraform)
│   ├── gateway.py                    # Creates MCP gateway + registers tool targets
│   └── schemas/                      # Same schemas referenced by gateway.py
│
├── agentcore_identity/               # Workload identity setup (run post-terraform)
│   └── create_workload_identity.py
│
├── UI/                               # React / TypeScript frontend
│   ├── src/
│   │   ├── api/                      # All AWS SDK / API calls abstracted here
│   │   ├── pages/                    # One folder per page
│   │   ├── components/               # Reusable UI components
│   │   └── context/                  # React context (auth state, agent invocation state)
│   ├── package.json
│   └── vite.config.ts
│
├── env_generate.py                   # Bulk SSM fetch → generates UI .env file
└── deploy.sh                         # Full end-to-end deployment script
```

## Naming Conventions

### The `project_name` variable

Every resource name in the entire stack is derived from one `project_name` Terraform variable. This is the **only thing you change** to spin up a new environment.

```
project_name = "retail-analytics-dev"
```

| Resource | Resulting name |
|----------|----------------|
| S3 data bucket | `retail-analytics-dev-<account-id>-data` |
| S3 athena-results bucket | `retail-analytics-dev-<account-id>-athena-results` |
| DynamoDB traces table | `retail-analytics-dev-agent-traces` |
| Lambda IAM role | `retail-analytics-dev-lambda-execution-role` |
| SSM parameter | `/retail-analytics-dev/data/bucket-name` |
| Cognito user pool | `retail-analytics-dev-m2m-user-pool` |
| AgentCore agent | `retail_analytics_dev_trend_analysis_agent` (hyphens → underscores) |
| Glue database | `retail_analytics_dev_database` (hyphens → underscores) |

> **Rule:** Never hardcode a resource name anywhere. Always derive it from `var.project_name` in Terraform, or read from SSM at runtime in Lambda / agent code.

### Deriving agent names in deploy.sh

```bash
PROJECT_NAME=$(cd terraform/ && terraform output -raw project_name)
# Convert hyphens to underscores for AgentCore (which requires snake_case names)
AGENT_TREND_NAME="$(echo $PROJECT_NAME | tr '-' '_')_trend_analysis_agent"
AGENT_ANOMALY_NAME="$(echo $PROJECT_NAME | tr '-' '_')_anomaly_detection_agent"
```

### Terraform file naming

Each `.tf` file owns exactly one concern. Add a new file rather than mixing concerns in `main.tf`.

| File | Owns |
|------|------|
| `data_infrastructure.tf` | S3, Glue catalog, DynamoDB tables, Athena config |
| `backend_lambdas.tf` | API-facing Lambda functions + their IAM roles |
| `agent_lambdas.tf` | Agent tool Lambda functions |
| `report_lambda.tf` | Async report-generation Lambdas (initiator, worker, status checker) |
| `report_ssm_parameters.tf` | All SSM parameters for report job runtime config |
| `api_gateway.tf` | API Gateway: API, routes, integrations, CORS, authorizer |
| `locals.tf` | **Only** computed locals — no resource blocks |
| `variables.tf` | **Only** variable declarations |
| `outputs.tf` | **Only** output values |

### SQL file naming

Use `snake_case` names that match the view they create:

```
terraform/sql/
├── monthly_sales_view.sql            # creates view: monthly_sales_view
├── quarterly_revenue_view.sql        # creates view: quarterly_revenue_view
├── anomaly_detection_view.sql        # creates view: anomaly_detection_view
└── trend_analysis_view.sql           # creates view: trend_analysis_view
```

## What Goes Where — Decision Table

| I have a... | Put it in |
|-------------|-----------|
| Lambda that serves an API endpoint (GET /products, GET /sales-summary) | `backend/` |
| Lambda that is a **tool** for a Bedrock agent | `agents/<agent-name>/tools/` |
| Lambda for async long-running processing (initiator / worker / status checker) | `report_lambda/` |
| Terraform resource definition | `terraform/<concern>.tf` |
| SQL view for Athena | `terraform/sql/` |
| Raw CSV or Parquet data file | `data/csv/` or `data/parquet/` |
| Bedrock agent entry-point code | `agents/<agent-name>/agent/` |
| JSON schema for an agent tool | `agents/<agent-name>/schemas/` AND `agentcore_gateway/schemas/` |
| Post-deploy Python script (gateway, identity) | root level alongside `deploy.sh` |
| UI component or page | `UI/src/` |
| Environment-specific runtime value | SSM Parameter Store — written by Terraform, read at runtime |

## The `data/` Folder — Why It Is Separate from `terraform/`

Data files (CSV, Parquet) are **not infrastructure**. Mixing them into the Terraform folder creates noise in `terraform plan` output and couples data refresh cycles to infrastructure deployments.

### Correct separation

```
data/
├── csv/                  ← source files, excluded via .gitignore if large
├── parquet/              ← generated by csv_to_parquet.py, also gitignored
└── csv_to_parquet.py     ← run once (or periodically) before terraform apply
```

Terraform then references the `parquet/` output with a relative path pointing one directory up:

```hcl
# terraform/data_infrastructure.tf
resource "aws_s3_object" "orders" {
  bucket = aws_s3_bucket.data.id
  key    = "${var.data_prefix}/orders/orders.parquet"
  source = "${path.module}/../data/parquet/orders.parquet"
  etag   = filemd5("${path.module}/../data/parquet/orders.parquet")

  depends_on = [aws_s3_bucket.data, aws_s3_bucket_policy.data_policy]
}
```

## Agent Folder — Sub-structure

Each agent has three sub-folders with strict separation of concerns:

```
agents/trend-analysis-agent/
├── agent/
│   ├── trend_analysis_agent.py  ← ONLY entry point + Strands orchestration logic
│   ├── requirements.txt         ← deps for the container image built by AgentCore
│   └── .bedrock_agentcore.yaml  ← generated by `agentcore configure`, do not edit manually
├── tools/
│   ├── get_sales_by_region.py   ← Lambda function code for each tool
│   └── get_trend_metrics.py
└── schemas/
    ├── get-sales-by-region.json  ← OpenAPI-style JSON schema describing each tool's input/output
    └── get-trend-metrics.json
```

The same schema JSON files appear in **both** `agents/<agent>/schemas/` (for the agent's reference) and `agentcore_gateway/schemas/` (read by `gateway.py` to register gateway targets).

## `.bedrock_agentcore.yaml` — What It Contains

This file is auto-generated by `agentcore configure`. Do not edit it manually, but do commit it so re-deployments are idempotent. Key fields:

```yaml
# agents/trend-analysis-agent/agent/.bedrock_agentcore.yaml
default_agent: retail_analytics_dev_trend_analysis_agent
agents:
  retail_analytics_dev_trend_analysis_agent:
    name: retail_analytics_dev_trend_analysis_agent
    entrypoint: /path/to/trend_analysis_agent.py
    deployment_type: container
    platform: linux/arm64
    aws:
      execution_role: arn:aws:iam::<account-id>:role/retail-analytics-dev-bedrock-agentcore-execution-role
      region: us-east-1
      ecr_repository: <account-id>.dkr.ecr.us-east-1.amazonaws.com/bedrock-agentcore-retail_analytics_dev_trend_analysis_agent
    authorizer_configuration:
      customJWTAuthorizer:
        discoveryUrl: https://cognito-idp.us-east-1.amazonaws.com/<pool-id>/.well-known/openid-configuration
        allowedClients:
          - <m2m-client-id>
          - <web-client-id>
```
